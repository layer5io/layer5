<Link to="/cloud-native-management/meshery">Meshery</Link> treats your ConfigMaps and Secrets as first-class design objects. You define them once in a visual canvas or a declarative design file, and Meshery propagates them to as many clusters as you tell it to—atomically, consistently, and with a complete audit trail. This post walks through the mechanics with real commands.

## Why ConfigMap and Secret Management Hurts at Scale

Before getting into the solution, it's worth naming the specific pain points that accumulate as your fleet grows.

**Drift.** Clusters diverge silently. A ConfigMap updated in one cluster never makes it to another. Services behave differently in ways that are extremely hard to debug because the configuration looks identical in your Git repository—it just isn't what's actually running.

**Manual toil.** Every cluster is a separate `kubectl` invocation. Scripting this with bash loops works until it doesn't: a failed API call leaves half your fleet updated, and there's no easy rollback.

**Secret sprawl.** Secrets stored as base64 in YAML files checked into Git. Secrets recreated manually in every namespace. Secrets that nobody knows how to rotate because the person who created them left the team.

**No visibility.** `kubectl get configmap` tells you what's in one cluster, right now. It tells you nothing about whether that matches what's in your other eleven clusters, or whether it matches what's in your Helm chart, or whether it matches what you deployed last Thursday.

## Setting Up Meshery for Multi-Cluster Work

If you haven't installed Meshery yet, the fastest path is through `mesheryctl`:

```bash
# Install mesheryctl
curl -L https://meshery.io/install | bash -

# Start Meshery (spins up local server and opens the dashboard)
mesheryctl system start

# Confirm Meshery is running
mesheryctl system status
```

Once Meshery is running, connect your clusters. Meshery reads standard kubeconfig files, so if you already have multiple contexts configured, you're most of the way there:

```bash
# Check what contexts Meshery sees
mesheryctl system context list

# If your clusters aren't showing up, point Meshery at your kubeconfig
mesheryctl system context create prod-us-east \
  --kubeconfig ~/.kube/config \
  --context prod-us-east-1

mesheryctl system context create prod-eu-west \
  --kubeconfig ~/.kube/config \
  --context prod-eu-west-1

mesheryctl system context create staging \
  --kubeconfig ~/.kube/config \
  --context staging-us-east-1
```

You can also upload kubeconfig files directly from the Meshery dashboard under **Settings → Environments → Kubernetes**. Meshery will enumerate the contexts and present each cluster as a manageable target.

## Designing a ConfigMap in Meshery

In Meshery, infrastructure is expressed as a **design**—a declarative, versioned description of Kubernetes resources. A design can contain anything: Deployments, Services, ConfigMaps, Secrets, or all of the above together.

### Via the CLI

Create a design file that defines your ConfigMap:

```yaml
# app-config-design.yaml
name: app-config
version: "1.0.0"
services:
  app-config:
    type: ConfigMap
    apiVersion: v1
    metadata:
      name: app-config
      namespace: production
    spec:
      data:
        DATABASE_HOST: "postgres.internal.example.com"
        DATABASE_PORT: "5432"
        LOG_LEVEL: "info"
        FEATURE_FLAGS: "new-checkout=true,dark-mode=false"
        MAX_CONNECTIONS: "100"
```

Import and deploy the design:

```bash
# Import the design into Meshery
mesheryctl design import -f app-config-design.yaml

# Deploy to a single cluster
mesheryctl design deploy -f app-config-design.yaml \
  --context prod-us-east

# Deploy to multiple clusters in one command
mesheryctl design deploy -f app-config-design.yaml \
  --context prod-us-east,prod-eu-west,staging
```

The `--context` flag accepts a comma-separated list of any contexts Meshery knows about. Meshery applies the design to all of them in parallel and reports per-cluster success or failure.

### Verifying Deployment

After deploying, verify the ConfigMap landed correctly on each cluster:

```bash
# Check via mesheryctl
mesheryctl design status app-config

# Or verify directly with kubectl against each context
kubectl --context prod-us-east-1 get configmap app-config -n production -o yaml
kubectl --context prod-eu-west-1 get configmap app-config -n production -o yaml
kubectl --context staging-us-east-1 get configmap app-config -n production -o yaml
```

## Managing Secrets Without the YAML Clipboard

Secrets follow the same design pattern, but Meshery adds an important layer: it integrates with external secret stores so you never put plaintext values in design files.

### Defining a Secret in a Design

For secrets you're comfortable managing through Meshery directly (for example, non-sensitive connection metadata or tokens with limited scope):

```yaml
# tls-secret-design.yaml
name: tls-secret
version: "1.0.0"
services:
  api-tls:
    type: Secret
    apiVersion: v1
    metadata:
      name: api-tls
      namespace: production
    spec:
      type: kubernetes.io/tls
      data:
        # Values should be base64-encoded; use SOPS or Sealed Secrets
        # for secrets that need to be stored in version control
        tls.crt: "$(TLS_CERT_B64)"
        tls.key: "$(TLS_KEY_B64)"
```

Deploy with environment variable substitution:

```bash
# Export the values (from your secrets manager, CI/CD system, etc.)
export TLS_CERT_B64=$(cat tls.crt | base64 -w 0)
export TLS_KEY_B64=$(cat tls.key | base64 -w 0)

# Meshery resolves $(VAR) references at deploy time
mesheryctl design deploy -f tls-secret-design.yaml \
  --context prod-us-east,prod-eu-west
```

### Using Sealed Secrets with Meshery Designs

If your team uses Sealed Secrets, your workflow integrates cleanly:

```bash
# Seal the secret once
kubectl create secret generic db-credentials \
  --from-literal=username=app_user \
  --from-literal=password="${DB_PASSWORD}" \
  --dry-run=client -o json \
  | kubeseal --format yaml > db-credentials-sealed.yaml

# Reference the sealed secret in your Meshery design
# and deploy it to all clusters—kubeseal handles decryption
# cluster-side using the cluster's private key
mesheryctl design import -f db-credentials-sealed.yaml
mesheryctl design deploy -f db-credentials-sealed.yaml \
  --context prod-us-east,prod-eu-west,staging
```

Note that Sealed Secrets requires the SealedSecrets controller to be running in each target cluster, and each cluster will have its own sealing key. For cross-cluster secrets, use `--scope cluster-wide` when sealing.

## Detecting and Resolving Configuration Drift

This is where Meshery earns its keep for platform engineers. After deploying a design, Meshery continuously monitors whether the live state of your clusters matches the desired state defined in the design.

```bash
# Show the current drift status for all deployed designs
mesheryctl design list --all

# Check a specific design against all its target clusters
mesheryctl design status app-config --verbose
```

The output shows you, per cluster, whether the resource matches the design, has drifted, or is missing entirely. When drift is detected, you have two options:

**Re-apply the design** (bring the cluster back in line with the design):

```bash
mesheryctl design deploy -f app-config-design.yaml \
  --context prod-eu-west \
  --overwrite
```

**Update the design** (if the drift was intentional and should become the new baseline):

```bash
# Pull the current live state into a new design version
mesheryctl design export \
  --context prod-us-east \
  --resource configmap/app-config \
  --namespace production \
  -o app-config-v2.yaml

# Review the diff
diff app-config-design.yaml app-config-v2.yaml

# Import the updated design
mesheryctl design import -f app-config-v2.yaml
```

## Namespace-Level Propagation Patterns

A common platform engineering pattern is to maintain a single "golden" ConfigMap and replicate it across multiple namespaces within a cluster, or across both namespaces and clusters. Here's how to express that in a Meshery design:

```yaml
# multi-namespace-config.yaml
name: shared-observability-config
version: "1.0.0"
services:
  observability-config-default:
    type: ConfigMap
    metadata:
      name: observability-config
      namespace: default
    spec:
      data:
        OTEL_EXPORTER_OTLP_ENDPOINT: "http://otel-collector.monitoring:4317"
        OTEL_SERVICE_NAMESPACE: "production"

  observability-config-payments:
    type: ConfigMap
    metadata:
      name: observability-config
      namespace: payments
    spec:
      data:
        OTEL_EXPORTER_OTLP_ENDPOINT: "http://otel-collector.monitoring:4317"
        OTEL_SERVICE_NAMESPACE: "production"

  observability-config-orders:
    type: ConfigMap
    metadata:
      name: observability-config
      namespace: orders
    spec:
      data:
        OTEL_EXPORTER_OTLP_ENDPOINT: "http://otel-collector.monitoring:4317"
        OTEL_SERVICE_NAMESPACE: "production"
```

```bash
# Deploy the same observability config to all namespaces across all prod clusters
mesheryctl design deploy -f multi-namespace-config.yaml \
  --context prod-us-east,prod-eu-west
```

## Rolling Out a Configuration Change

When you need to update a ConfigMap across your fleet—a new feature flag, a changed endpoint, a bumped log level—the workflow is:

```bash
# 1. Update the design file locally
vim app-config-design.yaml
# Edit: FEATURE_FLAGS: "new-checkout=true,dark-mode=true,beta-nav=true"

# 2. Validate the design before deploying
mesheryctl design validate -f app-config-design.yaml

# 3. Deploy to staging first
mesheryctl design deploy -f app-config-design.yaml \
  --context staging

# 4. Verify on staging
kubectl --context staging-us-east-1 \
  get configmap app-config -n production \
  -o jsonpath='{.data.FEATURE_FLAGS}'

# 5. Roll out to production
mesheryctl design deploy -f app-config-design.yaml \
  --context prod-us-east,prod-eu-west

# 6. Confirm consistent state across all clusters
mesheryctl design status app-config
```

Because the design file lives in Git alongside your application code, this change goes through your normal PR review process. The `mesheryctl design deploy` command in step 5 can be run from CI/CD—your GitOps pipeline triggers on merge to main and Meshery handles the multi-cluster fanout.

## Integrating with Your GitOps Pipeline

For teams using <Link to="/cloud-native-management/kanvas">Kanvas</Link> or a standard GitOps workflow, Meshery fits naturally into your pipeline. Here's a minimal GitHub Actions workflow:

```yaml
# .github/workflows/deploy-config.yml
name: Deploy ConfigMaps and Secrets

on:
  push:
    branches: [main]
    paths:
      - 'meshery/designs/**'

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install mesheryctl
        run: curl -L https://meshery.io/install | bash -

      - name: Configure Meshery contexts
        run: |
          mesheryctl system context create prod-us \
            --kubeconfig <(echo "${{ secrets.KUBECONFIG_PROD_US }}") \
            --context prod-us
          mesheryctl system context create prod-eu \
            --kubeconfig <(echo "${{ secrets.KUBECONFIG_PROD_EU }}") \
            --context prod-eu

      - name: Deploy changed designs
        run: |
          for design in meshery/designs/*.yaml; do
            mesheryctl design deploy -f "$design" \
              --context prod-us,prod-eu
          done

      - name: Verify deployment
        run: |
          mesheryctl design list --all
```

## What This Looks Like at Day 60

After you've been running this workflow for a couple of months, the operational picture changes noticeably:

- **Incident response is faster.** When something breaks, you `mesheryctl design status` to immediately see whether any cluster has drifted from the desired configuration. You eliminate the "wait, was this the same on all clusters?" question.

- **Onboarding is faster.** A new cluster joins the fleet by registering a kubeconfig with Meshery and running `mesheryctl design deploy --context new-cluster` for each design. It inherits the full configuration state of the fleet in minutes.

- **Audits are easier.** Every design has a version history. You can see exactly what was deployed to which cluster at what time—without digging through kubectl history or CI logs.

- **You stop writing bash loops.** The multi-cluster fan-out that used to live in a fragile shell script now lives in Meshery's deployment engine, which handles retries, partial failures, and status reporting properly.

## Conclusion

ConfigMap and Secret management across a multi-cluster Kubernetes fleet is a solved problem—it just requires the right tooling. <Link to="/cloud-native-management/meshery">Meshery</Link> gives platform engineers a design-first workflow where configuration is defined once, deployed consistently, and continuously reconciled. The fifteen-cluster YAML clipboard ritual becomes a single `mesheryctl design deploy` command.

The key takeaways:

- **Use designs as your source of truth**, not kubectl invocations. Designs are versioned, reviewable, and replayable.
- **Deploy with `--context`** to target multiple clusters in one command. Meshery handles the fanout and reports per-cluster status.
- **Use environment variable substitution** for secret values so design files stay clean and can live in version control.
- **Monitor for drift** with `mesheryctl design status`. Configuration drift is silent and expensive; automated detection is cheap.
- **Integrate with CI/CD** so ConfigMap changes go through your normal review process and deploy automatically on merge.

Your infrastructure deserves the same engineering rigor as your application code. Meshery makes that practical.
