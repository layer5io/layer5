## Why OpenTelemetry, and Why Now

The observability landscape fractured badly in the vendor lock-in era. Every APM vendor shipped its own SDK, its own agent, and its own wire format. Migrating between backends meant re-instrumenting every service. OpenTelemetry — the CNCF project merging OpenTracing and OpenCensus — ended that fragmentation. OTel provides a single, vendor-neutral SDK and a standardized wire protocol (OTLP) that any backend can consume: Jaeger, Tempo, Zipkin, Datadog, Honeycomb, and dozens more.

For Kubernetes-native stacks the combination of OTel auto-instrumentation, the OTel Collector, and a <Link to="/meshery">Meshery-managed</Link> deployment workflow collapses what used to be a multi-day project into a reproducible, version-controlled design that can be applied to any cluster.

<Callout type="info" title="OTel Signals in Scope">
  <p>OpenTelemetry covers three signal types: traces, metrics, and logs. This guide focuses on distributed <strong>traces</strong> — the signal most directly useful during active incident response. Metrics and log correlation are natural follow-on steps once traces are flowing.</p>
</Callout>

## Architecture Overview

A minimal OTel tracing stack on Kubernetes has three layers:

1. **Instrumented services** — applications that emit spans via the OTel SDK or the OTel Operator's auto-instrumentation injection.
2. **OTel Collector** — a cluster-local pipeline that receives spans over OTLP, processes them (batching, tail sampling, attribute enrichment), and exports to one or more backends.
3. **Trace backend** — a storage and query layer. This guide uses Jaeger as the backend because it is free, open-source, and deployable in-cluster without external dependencies.

Meshery provides the control plane that lets you design, deploy, and manage all three layers as a single versioned design — and redeploy it to a different cluster in minutes.

```
 Service A ──OTLP──┐
 Service B ──OTLP──┤   OTel Collector   ──OTLP──► Jaeger (backend)
 Service C ──OTLP──┘   (DaemonSet or               │
                         Deployment)               ▼
                                            Jaeger Query UI
                                            (Grafana Tempo etc.)
```

## Step 1 — Deploy the OTel Operator

The [OpenTelemetry Operator](https://github.com/open-telemetry/opentelemetry-operator) is the fastest path to auto-instrumentation. It watches for `Instrumentation` CRs and injects the appropriate language SDK as an init container into matching pods — no code changes required for Java, Python, Node.js, Go, .NET, and Ruby workloads.

Install cert-manager first (the operator requires it for webhook certificates):

```bash
kubectl apply -f https://github.com/cert-manager/cert-manager/releases/latest/download/cert-manager.yaml
kubectl rollout status deployment cert-manager -n cert-manager --timeout=90s
```

Then install the OTel Operator:

```bash
kubectl apply -f https://github.com/open-telemetry/opentelemetry-operator/releases/latest/download/opentelemetry-operator.yaml
```

Verify:

```bash
kubectl get pods -n opentelemetry-operator-system
# NAME                                        READY   STATUS    RESTARTS
# opentelemetry-operator-controller-...       2/2     Running   0
```

## Step 2 — Deploy Jaeger as the Trace Backend

For in-cluster development and staging, the Jaeger all-in-one deployment is sufficient. Production deployments should use the Jaeger Operator with a distributed storage backend (OpenSearch or Cassandra).

```bash
helm repo add jaegertracing https://jaegertracing.github.io/helm-charts
helm repo update
helm install jaeger jaegertracing/jaeger \
  --namespace observability \
  --create-namespace \
  --set allInOne.enabled=true \
  --set provisionDataStore.cassandra=false \
  --set storage.type=memory \
  --set query.enabled=false \
  --set collector.enabled=false \
  --set agent.enabled=false
```

Expose the Jaeger UI for local access:

```bash
kubectl port-forward svc/jaeger-query 16686:16686 -n observability
```

Open `http://localhost:16686` — you have a trace backend. It is empty for now; that changes in Step 4.

## Step 3 — Configure the OTel Collector with Meshery

The OTel Collector is the heart of the pipeline. Rather than deploying it with raw YAML, use Meshery to design and manage the Collector configuration as a reproducible, shareable design.

### 3a — Create the Collector Configuration

The Collector needs a configuration that defines receivers (where spans arrive), processors (what to do with them), and exporters (where they go).

Create a `ConfigMap` for the Collector pipeline:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: otel-collector-config
  namespace: observability
data:
  config.yaml: |
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318

    processors:
      batch:
        timeout: 1s
        send_batch_size: 1024
      memory_limiter:
        check_interval: 1s
        limit_mib: 512
      resource:
        attributes:
          - key: deployment.environment
            value: production
            action: upsert

    exporters:
      otlp/jaeger:
        endpoint: jaeger-collector.observability.svc:4317
        tls:
          insecure: true
      debug:
        verbosity: basic

    service:
      pipelines:
        traces:
          receivers: [otlp]
          processors: [memory_limiter, batch, resource]
          exporters: [otlp/jaeger, debug]
```

### 3b — Deploy the Collector as an OpenTelemetryCollector CR

With the OTel Operator installed, you can declare the Collector as a CR:

```yaml
apiVersion: opentelemetry.io/v1alpha1
kind: OpenTelemetryCollector
metadata:
  name: otel-collector
  namespace: observability
spec:
  mode: deployment
  config: |
    receivers:
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318
    processors:
      batch:
        timeout: 1s
        send_batch_size: 1024
      memory_limiter:
        check_interval: 1s
        limit_mib: 512
    exporters:
      otlp/jaeger:
        endpoint: jaeger-collector.observability.svc:4317
        tls:
          insecure: true
    service:
      pipelines:
        traces:
          receivers: [otlp]
          processors: [memory_limiter, batch]
          exporters: [otlp/jaeger]
```

Apply it:

```bash
kubectl apply -f otel-collector.yaml
kubectl rollout status deployment otel-collector-collector -n observability
```

### 3c — Save the Design in Meshery

Once the Collector is running, export the current state into a Meshery design so you can reproduce it:

```bash
mesheryctl design import -f otel-collector.yaml --name "OTel Collector - Production"
```

From Meshery's Kanvas designer you can now visually inspect the Collector deployment, annotate its relationships with the Jaeger backend and your application namespaces, and share the design with your team or publish it to the <Link to="/catalog">Meshery catalog</Link>.

<Callout type="note" title="Collector Modes">
  <p>The example above uses <code>mode: deployment</code>, which is appropriate for centralized collection. Use <code>mode: daemonset</code> if you prefer per-node collection (lower network overhead for high-volume clusters) or <code>mode: sidecar</code> for per-pod collection in security-sensitive environments.</p>
</Callout>

## Step 4 — Auto-Instrument Your Services

The OTel Operator's `Instrumentation` CR defines which SDK version to inject and where to send telemetry. Create one for each language in your stack:

```yaml
apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: otel-instrumentation
  namespace: default
spec:
  exporter:
    endpoint: http://otel-collector-collector.observability.svc:4318
  propagators:
    - tracecontext
    - baggage
    - b3
  sampler:
    type: parentbased_traceidratio
    argument: "1.0"
  java:
    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-java:latest
  nodejs:
    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-nodejs:latest
  python:
    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-python:latest
  go:
    image: ghcr.io/open-telemetry/opentelemetry-operator/autoinstrumentation-go:latest
```

Then annotate the pods you want instrumented. The operator handles injection automatically:

```yaml
# For a Java service
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service
  namespace: default
spec:
  template:
    metadata:
      annotations:
        instrumentation.opentelemetry.io/inject-java: "true"
```

```yaml
# For a Python service
apiVersion: apps/v1
kind: Deployment
metadata:
  name: inventory-service
  namespace: default
spec:
  template:
    metadata:
      annotations:
        instrumentation.opentelemetry.io/inject-python: "true"
```

After the rollout completes, the pods will each contain an init container that copies the SDK into the application container and sets the appropriate environment variables (`OTEL_EXPORTER_OTLP_ENDPOINT`, `OTEL_SERVICE_NAME`, etc.).

Verify injection:

```bash
kubectl describe pod -l app=order-service | grep -A5 "Init Containers"
# opentelemetry-auto-instrumentation-java:
#   Image: ghcr.io/open-telemetry/.../autoinstrumentation-java:latest
```

## Step 5 — Verify Traces in Jaeger

Generate some traffic against your services, then check the Jaeger UI:

```bash
# Generate load (replace with your actual service endpoint)
for i in $(seq 1 50); do
  curl -s http://order-service.default.svc/api/orders > /dev/null
done
```

Open `http://localhost:16686`, select a service from the dropdown, and click **Find Traces**. You should see traces with multiple spans — one per service hop — connected by the W3C TraceContext propagation headers (`traceparent`, `tracestate`) that the OTel SDK injects automatically.

  quote="Distributed tracing is not just debugging infrastructure — it is a shared understanding of how your system actually behaves, not how you think it behaves."

A healthy trace looks like nested spans, each with its own duration, status, and attribute set. When a request crosses service boundaries, child spans appear indented under the parent. The total trace duration is the wall-clock time your end user experienced. Individual span durations tell you where that time was spent.

## Step 6 — Add Tail Sampling for Production Scale

A sampling rate of `1.0` (100%) is fine for development but unsustainable at production request volumes. Tail sampling — making the keep/drop decision after a complete trace arrives — lets you retain 100% of error traces and slow outliers while dropping the noisy normal majority.

Add the tail sampling processor to your Collector config:

```yaml
processors:
  tail_sampling:
    decision_wait: 10s
    num_traces: 100000
    expected_new_traces_per_sec: 1000
    policies:
      - name: errors-policy
        type: status_code
        status_code:
          status_codes: [ERROR]
      - name: slow-traces-policy
        type: latency
        latency:
          threshold_ms: 1000
      - name: probabilistic-policy
        type: probabilistic
        probabilistic:
          sampling_percentage: 5
```

This policy retains all error traces, all traces over 1 second, and 5% of everything else. Adjust `sampling_percentage` based on your volume and storage budget.

<Callout type="warning" title="Tail Sampling Requires a Single Collector Instance Per Trace">
  <p>Tail sampling only works correctly when all spans for a given trace arrive at the same Collector instance. If you run multiple Collector replicas, you need a load balancer exporter in a first-tier Collector to route spans by <code>traceId</code> before they reach the tail-sampling Collectors. The OTel Collector's <code>loadbalancingexporter</code> is built for exactly this.</p>
</Callout>

## Troubleshooting Common Issues

### Spans Not Appearing in Jaeger

1. Check Collector logs: `kubectl logs deployment/otel-collector-collector -n observability`
2. Verify the `OTEL_EXPORTER_OTLP_ENDPOINT` env var in instrumented pods matches the Collector service address
3. Confirm network policy allows egress from application namespaces to `observability` on ports 4317/4318
4. Use the Collector's `debug` exporter temporarily to confirm spans arrive: check for `"ResourceSpans"` log lines

### Auto-Instrumentation Init Container Not Injecting

1. Confirm the `Instrumentation` CR is in the same namespace as the pod (or configure cross-namespace with `WATCH_NAMESPACE`)
2. Verify the annotation key matches exactly: `instrumentation.opentelemetry.io/inject-java` for Java, etc.
3. Check operator webhook logs: `kubectl logs deployment/opentelemetry-operator-controller-manager -n opentelemetry-operator-system`

### High Cardinality Attribute Explosions

OTel SDKs sometimes emit high-cardinality attributes (user IDs, request IDs) as span attributes. This strains trace backends. Add the `attributes` processor to drop or hash problematic keys:

```yaml
processors:
  attributes:
    actions:
      - key: user.id
        action: delete
      - key: http.request.id
        action: hash
```

## Operationalizing the Stack

Once traces are flowing, three operational habits will keep the stack healthy long-term:

**1. Alert on Collector drop rate.** The Collector exposes `otelcol_processor_dropped_spans` as a Prometheus metric. A non-zero value means spans are being dropped — usually because the `memory_limiter` is triggered. Alert at 1% drop rate, page at 5%.

**2. Version your Meshery designs.** Treat the OTel design like application code: tag releases, review changes in pull requests, and use `mesheryctl design diff` to catch unintended configuration drift.

**3. Test trace propagation in CI.** Add a smoke test to your pipeline that sends a synthetic request through your service graph and asserts a complete trace appears in the backend. Broken propagation is silent — services instrument themselves, but if header injection is missing in one middleware layer, the trace chain breaks.

## What Is Next

Distributed traces answer "where is the time going?" With that baseline in place, the natural extensions are:

- **Log correlation**: Inject `trace_id` and `span_id` into structured log lines using the OTel logging SDK so you can jump directly from a slow span to its corresponding log entries.
- **Metrics exemplars**: Attach `trace_id` references to Prometheus histograms so you can click from a latency spike on a dashboard directly to a representative trace.
- **Continuous profiling**: Tools like Parca or Pyroscope can attach CPU profiles to spans, giving you function-level attribution for latency without code instrumentation changes.

Each of these integrates naturally into the Meshery-managed design you built today. The OTel Collector is already in place; adding new pipelines is a configuration change, not a deployment.
