Managing Kubernetes ConfigMaps and Secrets across multiple clusters is one of those problems that feels solved until it isn't. `kubectl apply -f config.yaml` works fine for one cluster. For five clusters across three regions, manual synchronization is a reliability hazard, an audit nightmare, and a genuine waste of platform engineering talent.

This post walks through how to use <Link to="/cloud-native-management/meshery">Meshery</Link> to design configuration resources once, deploy them consistently across clusters, and keep them in sync — without maintaining a bespoke shell script graveyard.

## The Problem With Manual ConfigMap and Secret Management

Before getting into the solution, it's worth naming exactly what breaks down at scale.

**Configuration drift.** You update a ConfigMap in staging to debug an issue. You forget to backport the change. Prod now has different behavior. The incident happens three weeks later and takes four hours to root-cause.

**Secret sprawl.** Secrets get created manually with `kubectl create secret generic`, never committed to any design file, never tracked in Git. When a cluster is rebuilt, nobody knows what secrets existed. The runbook says "ask the person who set it up."

**No audit trail.** Who changed the `database-config` ConfigMap in the east region cluster last Tuesday? If the answer is "I'll check Slack," you don't have an audit trail.

**Toil.** Copying a ConfigMap to eight clusters looks like this:

```bash
for ctx in $(kubectl config get-contexts -o name); do
  kubectl --context $ctx apply -f configmap.yaml
done
```

That loop breaks when a cluster is unreachable. It doesn't validate that the apply succeeded. It doesn't tell you which clusters are now out of sync with the desired state. It is a maintenance liability disguised as automation.

Meshery treats your Kubernetes resources as a **Design** — a versioned, visual artifact that can be deployed to one or many targets in a single operation. Let's see what that looks like in practice.

## Prerequisites

- [Meshery installed](https://docs.meshery.io/installation) (`mesheryctl` available and connected to at least one cluster)
- Two or more Kubernetes clusters registered in Meshery
- `kubectl` configured with contexts for each cluster (Meshery will import these)

If you don't have Meshery installed yet:

```bash
curl -L https://meshery.io/install | PLATFORM=kubernetes bash -
mesheryctl system start
```

Then open `http://localhost:9081` in your browser. Meshery will prompt you to connect your clusters.

## Step 1: Connect Your Clusters

Meshery uses your local kubeconfig to discover available cluster contexts. To register a cluster:

```bash
mesheryctl system context create prod-east \
  --url https://your-prod-east-api-endpoint \
  --set
```

Or from the Meshery UI, navigate to **Lifecycle > Kubernetes** and click **Add Cluster**. Paste your kubeconfig or select a context from the dropdown. Repeat for each cluster.

Once registered, each cluster appears as a deployment target. When you deploy a Design, you choose which targets receive it — one cluster, a subset, or all of them at once.

## Step 2: Build a ConfigMap Design in Kanvas

<Link to="/kanvas">Kanvas</Link>, the visual design tool embedded in Meshery, lets you build Kubernetes resources on a canvas without writing raw YAML. Here's the practical flow for a ConfigMap.

1. Open **Kanvas Designer** from the Meshery dashboard.
2. Search for "ConfigMap" in the component panel and drag it onto the canvas.
3. Click the component to open the configuration panel.
4. Fill in your namespace, name, and key/value pairs.

For a database configuration ConfigMap, the underlying YAML would look like this — but you're editing it through a form, not a text editor:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: database-config
  namespace: platform
data:
  DB_HOST: "postgres.platform.svc.cluster.local"
  DB_PORT: "5432"
  DB_NAME: "appdb"
  DB_MAX_CONNECTIONS: "50"
  DB_IDLE_TIMEOUT: "300s"
```

<Callout type="tip" title="Design annotations">
  <p>Use the annotation field in Kanvas to document why a value exists, not just what it is. Future you — or the on-call engineer at 2 AM — will thank present you. Example: <code>DB_MAX_CONNECTIONS: "50" # Tuned for t3.medium nodes; increase if upgrading node class</code>.</p>
</Callout>

You can also import an existing ConfigMap from a live cluster directly into a Design:

```bash
mesheryctl design import \
  --file configmap.yaml \
  --name "platform-database-config"
```

Or, pull it live from a cluster and convert it:

```bash
kubectl get configmap database-config -n platform -o yaml | \
  mesheryctl design import --stdin --name "platform-database-config"
```

Save the Design. It is now version-controlled in Meshery and can be deployed to any registered cluster.

## Step 3: Handle Secrets Without Secrets in Your Design Files

Secrets deserve special treatment. You should never commit raw Secret manifests to Git — the base64 encoding in a Kubernetes Secret is not encryption, it's just encoding. Anyone with access to the repository can decode it in seconds:

```bash
echo "c3VwZXJzZWNyZXQ=" | base64 --decode
# supersecret
```

The right pattern is to store secrets in a secrets manager (Vault, AWS Secrets Manager, GCP Secret Manager, Azure Key Vault) and reference them at deploy time. Meshery supports this through its environment variables and connection configurations.

For the Kanvas design, you add a Secret component with placeholder values:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: database-credentials
  namespace: platform
type: Opaque
data:
  DB_PASSWORD: ${DB_PASSWORD_BASE64}
  DB_USERNAME: ${DB_USERNAME_BASE64}
```

The `${VAR}` references are resolved from Meshery's environment configuration at deploy time. You set these once per environment:

```bash
mesheryctl environment variable set DB_PASSWORD_BASE64 \
  --env production \
  --value "$(echo -n "$PROD_DB_PASSWORD" | base64)"
```

<Callout type="warning" title="Rotate secrets in Meshery, not just in Kubernetes">
  <p>When you rotate a secret, update it in Meshery's environment configuration and redeploy the Design. This ensures the change is tracked, all target clusters receive the update, and the new value is the authoritative source. Updating secrets directly with <code>kubectl edit secret</code> bypasses the Design and reintroduces drift.</p>
</Callout>

## Step 4: Deploy to Multiple Clusters at Once

This is where Meshery earns its keep for platform engineers. With your Design saved, deploying to multiple clusters is a single operation.

From the CLI:

```bash
mesheryctl design deploy \
  --design "platform-database-config" \
  --env production \
  --contexts prod-east,prod-west,prod-eu
```

Meshery will:

1. Resolve environment variable references for the `production` environment
2. Connect to each specified cluster context
3. Apply the ConfigMap and Secret manifests
4. Report success or failure per cluster

The output looks like:

```
Deploying "platform-database-config" to 3 contexts...
  ✔ prod-east   (configmap/database-config created, secret/database-credentials configured)
  ✔ prod-west   (configmap/database-config created, secret/database-credentials configured)
  ✗ prod-eu     (connection timeout — cluster unreachable)

2/3 deployments succeeded. Run 'mesheryctl design status platform-database-config' to check sync state.
```

One cluster failed. You know immediately. You don't discover it three weeks later during an incident.

From the Kanvas UI, you get a visual deployment dialog where you select target clusters with checkboxes and watch the deployment status update in real time per cluster.

## Step 5: Detect and Remediate Configuration Drift

Meshery continuously compares deployed resources against the desired state in your Designs. To check the current sync state:

```bash
mesheryctl design status "platform-database-config" --all-contexts
```

Sample output:

```
Design: platform-database-config (v3, last deployed 2026-03-28)

CONTEXT       RESOURCE                        STATUS     DRIFT
prod-east     configmap/database-config       In Sync    None
prod-east     secret/database-credentials     In Sync    None
prod-west     configmap/database-config       Drifted    DB_MAX_CONNECTIONS: "50" → "100"
prod-west     secret/database-credentials     In Sync    None
prod-eu       configmap/database-config       Unknown    Cluster unreachable
prod-eu       secret/database-credentials     Unknown    Cluster unreachable
staging       configmap/database-config       In Sync    None
staging       secret/database-credentials     In Sync    None
```

`prod-west` has drifted — someone ran `kubectl edit configmap` directly and changed `DB_MAX_CONNECTIONS` to `100`. You can see exactly what changed, when, and reconcile it:

```bash
# Reconcile prod-west back to the Design's desired state
mesheryctl design deploy \
  --design "platform-database-config" \
  --env production \
  --contexts prod-west \
  --force
```

Or you can update the Design to make `100` the new desired value, then deploy that updated version everywhere:

```bash
mesheryctl design edit "platform-database-config" \
  --set "configmap.database-config.data.DB_MAX_CONNECTIONS=100"

mesheryctl design deploy \
  --design "platform-database-config" \
  --env production \
  --contexts prod-east,prod-west,prod-eu
```

The design is now at version 4. Every change is versioned. You can diff any two versions:

```bash
mesheryctl design diff "platform-database-config" --from v3 --to v4
```

## Step 6: Namespace Templating for Multi-Tenant Clusters

If you run multiple tenants or environments on shared clusters, your ConfigMaps and Secrets often need to land in different namespaces per deployment target. Kanvas supports namespace templating in Designs:

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: database-config
  namespace: ${TARGET_NAMESPACE}
data:
  DB_HOST: ${DB_HOST}
  DB_PORT: "5432"
```

When deploying, pass namespace as an environment variable:

```bash
mesheryctl design deploy \
  --design "platform-database-config" \
  --env staging \
  --contexts staging-cluster \
  --set TARGET_NAMESPACE=team-alpha
```

Deploy the same Design to `team-beta` namespace on the same cluster:

```bash
mesheryctl design deploy \
  --design "platform-database-config" \
  --env staging \
  --contexts staging-cluster \
  --set TARGET_NAMESPACE=team-beta
```

One Design, multiple tenants, zero copy-pasting.

## Step 7: Integrate With Your CI/CD Pipeline

For most platform engineering teams, the goal is to remove human hands from the deployment loop entirely. Meshery has a REST API and GitHub Actions integration that makes this straightforward.

**GitHub Actions example — deploy on merge to main:**

```yaml
name: Deploy Platform Config

on:
  push:
    branches: [main]
    paths:
      - 'designs/**'

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install mesheryctl
        run: curl -L https://meshery.io/install | PLATFORM=linux bash -

      - name: Configure Meshery connection
        run: |
          mesheryctl system config \
            --url ${{ secrets.MESHERY_SERVER_URL }} \
            --token ${{ secrets.MESHERY_TOKEN }}

      - name: Deploy to staging
        run: |
          mesheryctl design deploy \
            --design "platform-database-config" \
            --env staging \
            --contexts staging-cluster

      - name: Verify staging deployment
        run: |
          mesheryctl design status "platform-database-config" \
            --contexts staging-cluster \
            --assert in-sync

      - name: Deploy to production
        if: success()
        run: |
          mesheryctl design deploy \
            --design "platform-database-config" \
            --env production \
            --contexts prod-east,prod-west,prod-eu
```

The `--assert in-sync` flag makes `mesheryctl design status` exit non-zero if any resources are drifted or unreachable. CI fails fast. Production gates on staging health.

<Callout type="note" title="Meshery token scoping">
  <p>Use scoped API tokens for CI/CD pipelines. Create a token with deploy-only permissions for the production contexts and a separate token for staging. This limits the blast radius if a CI secret is ever compromised. Tokens are managed at <strong>Lifecycle > Identity > API Tokens</strong> in the Meshery UI.</p>
</Callout>

## Visualizing Your Configuration Topology

One underappreciated feature of Kanvas is the Visualizer mode, which shows you live resources from all connected clusters on a single canvas. For ConfigMap and Secret management, this means you can see at a glance which clusters have which versions of your configuration resources, and which pods are consuming them.

Navigate to **Kanvas Visualizer**, select all your production cluster contexts, and filter by `kind: ConfigMap` or `kind: Secret`. You'll see a graph of every ConfigMap and Secret across your fleet, which workloads mount them as volumes or inject them as environment variables, and whether they match your Designs.

This view is particularly useful for answering questions like:
- "Is any workload in prod still referencing the old `legacy-db-config` ConfigMap we were supposed to retire?"
- "Which namespaces have the `tls-bundle` Secret and which are missing it?"

## Putting It Together: The Platform Engineer Workflow

The end state looks like this:

1. **Define once.** Create ConfigMap and Secret resources in a Meshery Design using Kanvas. Set environment-specific values through Meshery's environment configuration, not hardcoded in the Design.

2. **Deploy everywhere.** Run `mesheryctl design deploy` with the target contexts. Meshery handles the multi-cluster fan-out, resolves environment variables per target, and reports results per cluster.

3. **Detect drift automatically.** Meshery's continuous comparison catches manual changes. The `mesheryctl design status` command gives you a per-cluster, per-resource view of what's in sync and what's drifted.

4. **Reconcile or promote changes.** Either push the Design back to drifted clusters or update the Design to capture intentional changes, then deploy the new version.

5. **Gate CI/CD on sync state.** Use `--assert in-sync` in your pipeline to fail the build if staging is drifted before promoting to production.

## What This Doesn't Replace

Meshery's multi-cluster deploy is the right tool for propagating ConfigMaps and Secrets across clusters you own and operate. It complements but doesn't replace:

- **A secrets manager** (Vault, AWS Secrets Manager) — Meshery references secrets; it doesn't store them. Keep your secrets in a purpose-built secrets store.
- **RBAC policies** — Who can trigger a deploy from Meshery? Configure Meshery's own RBAC and ensure service accounts have least-privilege access in each cluster.
- **Secrets rotation tooling** — Automate rotation at the secrets manager level. Meshery handles the propagation; your rotation policy handles the lifecycle.
